# Build vs. Buy: Migration Tooling Landscape

Research spike for issue #20. Before building #18 (SQL lineage map) and #19 (query cost audit), and before consolidating fleet-wide Oracle/GCP tooling in #17, this document checks each planned piece against what mature tools already do well: dbt, SQLMesh, sqlglot, Google Dataform, BigQuery-native cost tooling, and OpenLineage/Marquez. The goal is to avoid quietly reimplementing a worse version of dbt's docs DAG or a worse version of sqlglot's transpiler — and, where a real gap exists, to be specific about why.

## The load-bearing distinction

Every comparison below turns on the same fact, already stated in `docs/architecture-rationale.md`: this repo is a **deterministic converter pipeline** for a **one-time, cross-dialect migration** of SQL nobody has vetted yet. dbt, SQLMesh, and Dataform are **ongoing, same-warehouse transformation tools** — they all assume the SQL already lives in one engine, is owned by the team running it, and is being iterated on over time. None of the three has any notion of translating unfamiliar SQL from a different engine, or of validating that a translation is *correct* against the untranslated original. That's not a minor omission; it's outside what the category of tool is for. Every "does dbt/SQLMesh/Dataform make this redundant?" question below resolves the same way for the same underlying reason.

## Verdicts on #17, #18, #19

**#18 (SQL lineage map): proceed as scoped.** dbt's, SQLMesh's, and Dataform's lineage graphs are all built from `ref()`/`source()` calls across a *persistent, ongoing project* — they answer "which of my maintained models depend on which," not "what did this one migration run touch, in what order, starting from tables nobody has modeled yet." None of them can ingest a raw legacy Oracle script with no pre-existing project structure and stage a lineage from scratch, because none of them do the translation step in the first place. This repo's per-run, dependency-staged markdown map is solving a problem those tools don't address, not a worse version of a problem they already solve. sqlglot's AST parsing (which SQLMesh's own lineage is built on) is a plausible *implementation detail* worth borrowing for source/target extraction robustness, but that's a `src/sql_processing.py` question, not a reason to drop #18.

**#19 (query cost audit): proceed as scoped, stay thin.** BigQuery's own dry-run (`total_bytes_processed`) is exactly the right primitive for estimating cost on freshly translated, not-yet-executed queries — #19 should be a thin wrapper around it (dry-run loop + a fixed on-demand/flat-rate pricing formula + markdown formatting), not a custom cost engine. Two things #19 must explicitly *not* rebuild: a historical cost tracker (`INFORMATION_SCHEMA.JOBS_BY_PROJECT` already does that, and it's the right tool once these queries have run for real) and BigQuery's own partitioning/clustering recommender (`INFORMATION_SCHEMA.RECOMMENDATIONS`), which is a real Google-maintained feature — it just structurally cannot fire for day-zero queries with no accumulated execution history. That gap (optimization suggestions for SQL that has never run in production yet) is the genuine, un-duplicated value of #19's LLM pass.

**#17 (fleet-wide Oracle/GCP tooling consolidation): unaffected by this research, proceed as scoped.** #17 is about consolidating real Oracle/BigQuery *client* code (auth, connection, query execution, upload/download) scattered across sibling repos — it doesn't overlap with any of the migration-specific tools compared here. The one adjacent finding: if `src/translator.py`'s deterministic fallback adopts sqlglot (recommended below), that dependency should be added once, in the consolidated location #17 is creating, not per-repo.

## Where to lean on an existing tool instead of hand-rolling

**Adopt sqlglot for the deterministic fallback's SQL-expression rewrites.** sqlglot officially supports `transpile(sql, read="oracle", write="bigquery")` today, and its Oracle dialect module already handles `NVL`, `TRUNC(date)`, `CREATE TABLE AS`, the `(+)` outer-join marker, `ROWNUM`/`ROWID`/`LEVEL`, Oracle-specific types, and more — a strict superset of the three hand-rolled rules `src/translator.py` implements now. It's maintained by Tobiko Data (the SQLMesh team), has no external dependencies of its own, and is already the transpilation engine underneath SQLMesh's lineage and dbt's dialect tooling. Swapping the fallback's regex rules for `sqlglot.transpile()` calls is close to a drop-in replacement and immediately covers far more of Oracle's SQL-expression surface than this repo will hand-write rules for over time.

**The honest gap sqlglot leaves — and why the LLM step still exists.** sqlglot is a SQL-expression transpiler, not a PL/SQL interpreter: it explicitly falls back to an opaque `Command` node for procedural constructs (`DECLARE` blocks, control flow, `BULK COLLECT`/`FORALL`, stored procedures). That's precisely the job `src/sql_processing.py` and the LLM step do today — variable materialization, statement splitting/ordering, and judgment calls on ambiguous rewrites — and nothing in this survey (sqlglot, SQLMesh, dbt, or Dataform) replaces that. Adopting sqlglot narrows the fallback's SQL-expression gap; it does not change the pipeline's shape.

**Don't adopt OpenLineage/Marquez now, but leave the door open.** OpenLineage's Python client is lightweight enough to emit lineage events without running a Marquez server, so a future `lineage.md` *and* an OpenLineage-shaped JSON facet from the same `RunReport.units` walk would be cheap to add later. But the interoperability payoff is speculative today: none of Airflow, Dagster, dbt, or a data catalog exist anywhere in this fleet to consume those events, and even dbt's own OpenLineage integration is an unofficial wrapper script, not native support. Keep the bespoke markdown map as the only lineage artifact until (if ever) another lineage-consuming tool actually shows up in the fleet.

**Dataform is not a build-vs-buy question for #17/#18/#19 — it's a possible future consumer.** Once this pipeline lands SQL on BigQuery, restating that output as Dataform SQLX models could be a reasonable next step for the *post-migration steady state* (scheduling, assertions, auto-docs, an ongoing lineage view) — but that is a distinct, later idea about what happens to the pipeline's output after migration, not something that changes how #18 or #19 should be scoped now.

## What would make this 10x better, not just a dbt clone

The tools above are all excellent at "you already own the SQL, on one warehouse, and want to iterate on it safely." None of them do cross-dialect migration validation at all — that gap is this repo's actual competitive surface, and it's worth investing in disproportionately rather than treating as a checkbox:

- **Ship the validation/repair loop and its audit trail as the headline feature, not a side effect.** dbt/SQLMesh/Dataform all assume the SQL is already trustworthy. This repo's dual-execution fingerprint comparison plus bounded, logged repair attempts is the one thing none of them offer — a translated query is provably equivalent to the original, not just syntactically plausible. The trace JSON (`run_trace_<timestamp>.json`) is already close to being a migration-specific audit product in its own right; making it a first-class, reviewable artifact (not just a debug dump) would be a genuine differentiator worth real UI/reporting investment.
- **Turn the schema compatibility audit into a pre-migration risk score, not just a pass/fail gate.** dbt/SQLMesh have nothing like it. Ranking unmapped/incompatible tables by how much downstream translation work they block (using the same lineage staging from #18) would make the audit a planning tool for the whole migration project, not just a per-table gate.
- **Feed #19's cost findings back into the repair loop.** If a translated query passes correctness validation but is flagged as unusually expensive by the dry-run estimate, that's a case worth an automatic second LLM pass proposing a rewrite before the migration is considered "done" — genuinely combining two planned features (#18's lineage staging for blast-radius, #19's cost estimate for triage) into a capability no single tool here offers.
- **Make the migration audit trail exportable in a shape other tools can consume**, once there's an actual second consumer in the fleet (see OpenLineage above) — the differentiator isn't the export format, it's that no other tool here produces a migration audit trail worth exporting in the first place.

## Recommendation

Proceed with #18 and #19 as currently scoped — neither duplicates dbt, SQLMesh, Dataform, or BigQuery-native tooling; both address a gap those tools structurally cannot fill (per-run migration lineage, and cost estimation for queries with no execution history). Separately, open a small follow-up to swap `src/translator.py`'s hand-rolled dialect rules for `sqlglot.transpile()` calls — that's a real, low-risk quality improvement, but it is *not* a reason to change #18/#19's scope or to reconsider the pipeline's overall shape. No other adoption/integration work is recommended at this time; OpenLineage emission and a Dataform-shaped output target are worth a one-line note in a future issue if the fleet's needs change, not action now.
